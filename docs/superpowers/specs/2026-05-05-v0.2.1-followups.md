# v0.2.1 Follow-ups (post-review)

**Source:** Three parallel reviews (security, Swift correctness, release pipeline) on 2026-05-05 caught a number of issues during the v0.2.0 phase. The user-facing must-fixes landed in v0.2.0 (cache hardening, supply-chain pin, timestamp fallback, humanTurns dedup, tick perf). Everything below is real but didn't block v0.2.0.

This is a working list, not a single-PR plan. Each item can ship independently.

---

## Pipeline robustness

### P1. `notarytool submit --wait` has no timeout
- **Where:** `.github/workflows/release.yml`, the "Notarize + staple" step
- **Why it matters:** We saw 2+ hr stalls during local testing. GH Actions' default job timeout is 360 min. If notarytool stalls past that, the job is force-killed mid-step, leaving notarization possibly successful in Apple's records but no stapling, no DMG published, no appcast updated. The `rm "$RUNNER_TEMP/key.p8"` also doesn't run.
- **Fix:** Add `--timeout 30m` to the notarytool invocation. Split notarize and staple into separate steps. Move secret tempfile cleanup into its own `if: always()` step (or a `trap`).

### P2. Secret tempfiles aren't cleaned up on failure paths
- **Where:** `.github/workflows/release.yml`, three steps decode secrets to `$RUNNER_TEMP/...` and `rm` at the end of the same `run:` block: `cert.p12`, `key.p8`, `sparkle_priv.txt`
- **Why it matters:** If any preceding command in the block fails, the `rm` never runs. Decoded signing material persists for the rest of the job. The runner VM is destroyed at job end, but defense in depth.
- **Fix:** Wrap each decode in `trap 'rm -f "$RUNNER_TEMP/cert.p12"' EXIT` inside the `run:` block, OR add `if: always()` cleanup steps at the end with explicit `rm -f` for each known tempfile.

### P3. `release.sh` cannot detect notarization-blockers before tagging
- **Where:** `scripts/release.sh:26-72` (preflight)
- **Why it matters:** Spent hours debugging notarization failures during v0.2.0 setup. The current preflight only checks git state + version + release-notes stanza. Doesn't catch entitlements regressions, hardened-runtime drift, new Sparkle binaries that need re-signing, etc.
- **Fix:** Add a `preflight_build()` phase that does a full local Release-config xcodebuild + `bash scripts/lib/sign_app.sh` + `codesign --verify --deep --strict` using ad-hoc identity (`CODE_SIGN_IDENTITY=-`). Catches ~80% of notarization failures in 60 seconds locally instead of 8 minutes in CI.

### P4. Mid-job `git checkout main` can race with concurrent commits
- **Where:** `.github/workflows/release.yml`, "Commit appcast.xml back to main" step
- **Why it matters:** If anyone pushes to `main` between the workflow's `git pull` and `git push`, the push is rejected and the job fails after notarization succeeded.
- **Fix:** (a) `git pull --rebase` with retry, OR (b) host the appcast off-main (GH Pages or release asset) so we don't need the commit-back-to-main dance at all. Option (b) is the better long-term answer — see P5.

### P5. `appcast.xml` lives on `main`; depends on `main` never being force-pushed
- **Where:** `Pits/Info.plist:25-26` (`SUFeedURL`)
- **Why it matters:** End users fetch from `raw.githubusercontent.com/joelisfar/pits/main/appcast.xml`. If `main` is ever force-pushed (intentionally or by accident), the appcast can disappear or revert. Older Pits clients hitting a rolled-back appcast see "no update available" and stay on the old version. Combined with key-compromise scenarios this is a downgrade attack vector.
- **Fix:** (a) Enable branch protection on `main` (no force-push, no deletion). (b) Long-term: move the appcast to GitHub Pages or a release asset, both append-only.

### P6. `sign_app.sh` hardcodes the Sparkle nested-binary list
- **Where:** `scripts/lib/sign_app.sh:31-35`
- **Why it matters:** If a future Sparkle version (2.10, 3.0) adds a new helper or XPC service, the script won't sign it. `codesign --verify --deep --strict` may pass anyway (`--deep` doesn't catch a missing signature on something not in the bundle's embedded structure metadata) but notarization will reject. With Sparkle now pinned to exact 2.9.1 (v0.2.0 fix), this is contained — but matters at every Sparkle bump.
- **Fix:** After signing the known list, run `find "$SPARKLE/XPCServices" -type d -name "*.xpc"` and `find "$SPARKLE" -maxdepth 2 -type f -perm +111` and assert no unknown executables exist. Forces a human to look when Sparkle changes structure.

### P7. `--preserve-metadata=identifier,entitlements,flags` not used on Sparkle nested re-sign
- **Where:** `scripts/lib/sign_app.sh:26`
- **Why it matters:** `codesign --force --sign` without `--preserve-metadata=entitlements` strips Sparkle's existing entitlements (XPC services have specific ones). Empirically v0.2.0 notarized fine without this, but it's a fragile contract — Apple may tighten enforcement, and Sparkle may rely on those entitlements at runtime.
- **Fix:** Add `--preserve-metadata=identifier,entitlements,flags` to the inner-binary loop. Keep the main-app re-sign line as-is (it has explicit `--entitlements`).

### P8. No verification that Sparkle public/private keys match
- **Where:** No CI step exists
- **Why it matters:** If keys are ever rotated and only one of `SUPublicEDKey` (in Info.plist) or `SPARKLE_ED_PRIVATE_KEY` (GH secret) gets updated, every running client thinks updates are corrupt and silently fails to update. Not detectable from CI.
- **Fix:** Add a CI step that derives the public key from the private key and asserts it matches `SUPublicEDKey` from the built `Pits.app/Contents/Info.plist`.

### P9. No `concurrency:` block on the workflow
- **Where:** `.github/workflows/release.yml`
- **Why it matters:** Two near-simultaneous tag pushes spawn two parallel CI jobs, both fighting for notarization slots and trying to push to `main`.
- **Fix:** Add `concurrency: { group: release, cancel-in-progress: false }` at workflow top level.

### P10. No stapler validation before publishing
- **Where:** `.github/workflows/release.yml`, after the staple step
- **Why it matters:** `xcrun stapler staple` can "succeed" with a partial result in unusual cases. Could ship a DMG that's notarized but lacks a working ticket.
- **Fix:** Add `xcrun stapler validate "$DMG"` and `spctl -a -t open --context context:primary-signature -vv "$DMG"` before publishing.

### P11. `gh release create` runs after appcast push (out-of-order failure mode)
- **Where:** `.github/workflows/release.yml`
- **Why it matters:** If the very last step (release create) fails, the appcast on `main` is already pointing at a Pits-X.Y.Z.dmg URL that doesn't exist yet. End users hitting Sparkle in that window get a 404.
- **Fix:** Swap order — publish the GH Release before pushing the appcast.

### P12. Atomic tag push
- **Where:** `scripts/release.sh:99`
- **Why it matters:** `git push origin main "v$VERSION"` — if `main` push succeeds but the tag push fails, main has the bumped commit but no tag, so CI doesn't trigger.
- **Fix:** `git push --atomic origin main "v$VERSION"`.

## Swift correctness

### S1. `LogWatcher.deinit` calls `stop()` without queue synchronization
- **Where:** `Pits/Services/LogWatcher.swift:42-43, 103-110`
- **Why it matters:** `stop()` mutates `self.stream` outside `queue.sync`. Under load (rapid file events on quit), could read a non-nil `stream` after `Invalidate` released it, or double-invalidate. Manifestation: rare crash on app quit.
- **Fix:** Wrap the body of `stop()` in `queue.sync { ... }`. Make `stop()` idempotent.

### S2. `Pricing.table` static state can be mutated from multiple actors
- **Where:** `Pits/Models/Pricing.swift:45-58`
- **Why it matters:** Currently safe in production because all callers happen to be on `MainActor`, but it's a global mutable static with no enforced isolation. Tests using `Pricing.replaceTable(with:)` via `defer` will race if `XCTestParallelize` is ever enabled. Future code paths could break the invariant invisibly.
- **Fix:** Annotate `Pricing` itself `@MainActor` (smallest change), or back the table with an `OSAllocatedUnfairLock`, or inject pricing into the store as a non-static dep.

### S3. `Conversation.projectName(from:)` does synchronous filesystem I/O during snapshot rebuild
- **Where:** `Pits/Models/Conversation.swift:182-198`, called from `ConversationStore.rebuildSnapshot`
- **Why it matters:** `rebuildSnapshot()` runs on `@MainActor`. For each conversation it walks parts on disk via `fm.fileExists(atPath:)`. With 1000+ JSONL files that's 1000+ stat syscalls on the main thread per rebuild. UI hangs / beach-ball under load.
- **Fix:** Cache `projectName` per file URL in `ConversationStore.fileBySession` (or sibling dict). Compute once when a sessionId/file pair is first seen, reuse on subsequent rebuilds.

### S4. `LogWatcher.readNewBytes` reads file growth into memory with no upper bound
- **Where:** `Pits/Services/LogWatcher.swift:200`
- **Why it matters:** On first backfill of a brand-new install, every `.jsonl` in `~/.claude/projects/**` is read into memory before line-splitting. No cap. On heavy users (100+ sessions × 100 MB each), or with a malicious JSONL containing a multi-GB single line, the app OOMs.
- **Fix:** Cap reads to e.g. 16 MB per call; loop with chunk-and-line-split. Cap any single line to e.g. 4 MB.

### S5. `OpenSessionsWatcher.openSessionIds()` may misread on concurrent rewrites
- **Where:** `Pits/Services/OpenSessionsWatcher.swift:33-37`
- **Why it matters:** `Data(contentsOf:)` on a file Claude Code is currently rewriting can read a truncated file. The decode then fails and that session falsely appears closed for one tick. With chime-once-per-window semantics, a one-tick miss at the threshold means the warning never fires for that window.
- **Fix:** Treat parse-failure-but-file-exists as "still open" by retaining the previous tick's set as fallback. Or read 2x with a 5ms gap and accept either parse.

### S6. Directory enumerators may follow symlinks outside the watched root
- **Where:** `Pits/Services/LogWatcher.swift:138-142`, `Pits/Stores/ConversationStore.swift:150-157`
- **Why it matters:** `FileManager.enumerator(at:)` follows symlinks by default. If anyone plants a symlink under `~/.claude/projects/` pointing elsewhere, the enumerator descends. Pits only opens `.jsonl` files, so impact is reading any `.jsonl` reachable through the link — defense in depth, not a real bug today.
- **Fix:** After resolving each enumerated URL, verify it still has the resolved root as a prefix. The `discoverFiles` rebase logic already computes both paths.

### S7. `RemotePricing` has no response-size cap
- **Where:** `Pits/Services/RemotePricing.swift:20-25`
- **Why it matters:** A malicious upstream (or MitM compromising TLS) could feed gigabytes within the 5s timeout window. `JSONSerialization.jsonObject(with:)` then parses it whole.
- **Fix:** Reject if `data.count > 1_048_576` (real file is ~150 KB).

### S8. `MonthScope.dateRange()` force-unwraps
- **Where:** `Pits/Models/MonthScope.swift:19-22, 30`
- **Why it matters:** A corrupted on-disk cache with `{"year":1,"month":99}` would survive Codable decoding and crash on first dateRange call.
- **Fix:** Validate in MonthScope's init, or use `?? Date()` fallback.

## Security

### Sec1. `os_log` uses `%{public}@` for error strings
- **Where:** `Pits/Services/RemotePricing.swift:27`, `Pits/Services/SnapshotCache.swift:60`
- **Why it matters:** `String(describing: error)` for a Foundation error often embeds the file URL. Unredacted in unified log → leaks paths in sysdiagnose / Console.
- **Fix:** Use `%{private}@` for error-description args, or log `error.localizedDescription` after sanitizing.

### Sec2. `projectName(from:)` walks attacker-influenceable paths via `fileExists`
- **Where:** `Pits/Models/Conversation.swift:167-199`
- **Why it matters:** Splits parent dir name on `-` and probes `fileExists` against `/`-prefixed paths. Worst case: render `passwd` (or whatever leaf matched) as the row's project name. No additional privilege gained beyond what reading JSONL bytes already provides.
- **Fix:** Constrain walk to `$HOME` prefix, or just always return `naiveLeaf`.

## Quality of life

### Q1. Move shared release-notes parser to a sourced shell function
- **Where:** `scripts/release.sh:68` and `.github/workflows/release.yml:90-94` parse `RELEASE_NOTES.md` with different patterns.
- **Why it matters:** Format drift risk if one place is updated and the other isn't.
- **Fix:** Move parser to `scripts/lib/extract_release_notes.sh` and source from both.

### Q2. Cron-style check for notarization key rotation / Sparkle URL liveness
- **Why it matters:** If the API key is revoked or `appcast.xml` becomes unreachable (force-push, repo rename), there's no signal until the next release fails or users complain.
- **Fix:** Weekly workflow that hits the appcast URL and runs `xcrun notarytool history`. Pings on failure.

### Q3. Document rotation runbooks
- **Why it matters:** Cert rotation, API key rotation, and Sparkle EdDSA key rotation are each rare enough to be forgotten.
- **Fix:** One-page runbook per credential. Add to `docs/`.

### Q4. Eliminate `KEYCHAIN_PASSWORD` GH secret
- **Where:** `.github/workflows/release.yml:24`
- **Why it matters:** Temp keychain on an ephemeral runner doesn't need a strong password — it's destroyed at job end.
- **Fix:** Generate inline: `KEYCHAIN_PASSWORD=$(openssl rand -hex 32)` and stop tracking as a secret.

## Things deliberately NOT in this list

- `LSUIElement` (Pits showing in Dock) — UX preference, not a defect
- BSD `sed -i ''` portability — script is mac-only, no real risk
- `actions/checkout@v4` SHA-pinning — first-party action, very low risk
- Sparkle-IPC dynamic instrumentation — separate audit if ever wanted
- Refactoring `ConversationStore` into smaller types — works as-is, no urgency

## Suggested ordering

If picking three to do next:
1. **P1 + P2** (notarytool timeout + secret cleanup on failure) — bundle, ~30 min
2. **S2** (Pricing thread-safety) — `@MainActor` annotation, ~10 min
3. **S1** (LogWatcher deinit) — `queue.sync` wrapper, ~5 min

If doing the supply-chain hardening pass:
4. **P3** (preflight build) — biggest payoff for time spent debugging future notarization issues
5. **P5** (move appcast off main) + **P9** (concurrency block) — together address most "weird CI failure" surface

Everything else can wait or be done opportunistically.
